# Workflow Builder

A visual, AI-assisted workflow editor for building, analyzing, and collaborating on process flows. Workflows are represented as directed acyclic graphs (DAGs) with typed nodes and edges. The builder includes an AI chat co-pilot, real-time collaboration, workflow analytics, and governance policy enforcement.

Product specs: [`Docs/workflow-v2.md`](../../../Docs/workflow-v2.md), [`Docs/workflow-analytics.md`](../../../Docs/workflow-analytics.md), [`Docs/workflow-builder-collaboration-features.md`](../../../Docs/workflow-builder-collaboration-features.md), [`Docs/ai-workflow-builder.md`](../../../Docs/ai-workflow-builder.md)

---

## What it does

- **Canvas**: drag-and-connect visual DAG editor with typed nodes (task, decision, trigger, AI, human) and edges (handoff types, channels)
- **AI Co-pilot (WorkflowChat)**: chat interface where users describe a process in natural language; the AI generates workflow nodes and edges, supports voice input via Whisper transcription, and responds with text-to-speech
- **Workflow Gallery**: browse and open saved workflows; create from templates
- **Collaboration Panel**: real-time presence indicators showing which team members are viewing the same workflow; live chat sidebar
- **Workflow Analytics**: metrics for a selected workflow — node count, cycle time, handoff efficiency, bottleneck identification
- **Governance enforcement**: workflow nodes flagged by the active governance policy are sent to the Approval Queue before being applied
- **History / undo**: automatic checkpoints after AI edits and manual edits (with configurable debounce); supports undo/redo

## Architecture & data flow

```
app/dashboard/governance-studio/page.tsx
    │
    └─► WorkflowBuilder.tsx
            │
            ├─► WorkflowsGallery     (open / create a workflow)
            │
            ├─► Canvas (SVG/DOM)     (drag, connect, edit nodes/edges)
            │
            ├─► WorkflowChat
            │       ├── POST /api/transcribe       (voice → text)
            │       ├── POST /api/text-to-speech   (text → audio)
            │       ├── POST /api/ai-builder        (text → workflow nodes/edges)
            │       └── POST /api/workflow-extract  (extract steps from free text)
            │
            ├─► WorkflowCollaborationPanel
            │       └── lib/workflowCollaboration/useWorkflowCollaboration.ts
            │               └── Supabase Realtime (presence + chat messages)
            │
            ├─► WorkflowAnalytics    (derived from workflow node/edge data)
            │
            └─► lib/workflowStorage/supabase.ts
                    └── Supabase `workflows` table (save, load, delete)
```

## Data

| Storage | What |
|---|---|
| Supabase `workflows` table | Workflow node/edge payload, metadata (name, owner, shared_with, revision, updated_at) |
| Supabase Realtime | Presence channel per workflow ID (who is viewing) + collaboration chat messages |
| `localStorage` | Chat history per workflow (`workflow-history:<id>`), user session ID |
| Governance policy | Read from `lib/governanceStorage.ts` (localStorage) for policy enforcement |

## Key design decisions

- **Debounced history checkpoints**: manual edits are committed to history after 3 seconds of inactivity (`MANUAL_EDIT_DEBOUNCE_MS = 3000`) to avoid creating a checkpoint per keystroke
- **Governance at the node level**: each node generated by the AI is checked against the active policy; flagged nodes are held in an approval queue rather than applied immediately
- **Layout engine**: `lib/workflowLayout.ts` (Dagre) is called to auto-arrange nodes after AI-generated additions; incremental layout is used for smaller changes to avoid jarring full redraws
- **Sensitivity classifier**: `lib/sensitivityClassifier.ts` scans node labels and descriptions for sensitive content before the AI processes them

## Current status & known gaps

- The canvas is a custom DOM/SVG implementation — not using a dedicated graph library like React Flow; this makes some interactions more complex to maintain
- Collaboration presence uses Supabase Realtime channels but does not yet support concurrent editing conflict resolution
- Workflow Analytics metrics are derived client-side from node/edge counts — not from actual execution data

---

## How to extend

- **Add a new node type**: extend the `WorkflowNode` type in `types/workflow.ts`, add rendering logic in the canvas section of `WorkflowBuilder.tsx`, and update `ai-builder` route prompts to generate the new type
- **Add execution tracking**: create a `workflow_runs` Supabase table, and update `WorkflowAnalytics.tsx` to query real execution data
- **Improve conflict resolution**: use Supabase Realtime broadcast events to sync node position changes across sessions in real time
